### 1. **사이트 목적 및 대상 사용자**

- **사이트 목적:**
  - 이 사이트는 온라인 가차 뽑기 참여자들이 ‘**불투명한 과정에서 오는 불안감 해소하기**’를 목표로, 누구나 직접 검증 가능한 **공정한 추첨(래플) 플랫폼**을 제공합니다.
  - 주최자가 “믿어주세요”가 아닌 “직접 확인해보세요”라고 말할 수 있는 구조
  - 참여자는 조작 의심 없이 “운이 없었다”고 받아들일 수 있는 **공정성 중심의 경험**
- 👥 대상 사용자

| 유형                   | 설명                                                            |
| ---------------------- | --------------------------------------------------------------- |
| **가치 지향형 소비자** | 공정한 경쟁과 투명한 프로세스에 매력을 느끼는 사용자            |
| **경험 추구형 사용자** | 단순 구매보다 ‘추첨 참여’ 자체를 즐기는 사용자                  |
| **신뢰 중시형 사용자** | SNS 기반의 불투명한 추첨보다 검증 가능한 시스템을 원하는 사용자 |

### 2. **주요 기능 및 페이지 구성**

- **필수 페이지:**
  - **홈 (Landing Page):** 방문자에게 첫인상을 주고 핵심 메시지를 전달하는 페이지. 기존 온라인 가차 시스템의 불투명한 서비스에 대한 문제제기를 하며 우리는 투명한 추첨 시스템을 기반으로 가차 시스템을 제공한다는 것을 어필할 것.
    - 모던한 UI/UX 인상을 사용자에게 줄 수 있도록 한다.
    - 온라인 가차 서비스를 사용하는 자들의 특성을 분석하여 이들의 시선을 사로잡을 수 있는 매력적인 UI/UX를 만든다.
    - 이러한 목적에 맞는 이미지와 아이콘들을 저작권이 없는 것들을 찾아 만들 것.
  - **로그인:** 사용자가 카카오 로그인으로 접속할 수 있는 화면

| 구분                  | 기능                                   | 설명                                          |
| --------------------- | -------------------------------------- | --------------------------------------------- |
| **추첨 등록**         | 운영자가 상품, 참여조건, 마감시간 등록 | 상품명, 이미지, 참여기간, seed 생성 기준 등록 |
| **추첨 참여**         | 회원이 로그인 후 참여                  | 1인 1회 제한, 중복방지 로직 적용              |
| **시드 공개**         | 추첨 전·후 외부 데이터 공개            | 예: KOSPI 종가, 날씨 데이터 등 외부 API       |
| **결과 생성 및 검증** | seed + 참여자 명단으로 난수 생성       | 코드 오픈소스로 공개, 누구나 재현 가능        |
| **회원 관리**         | 회원가입/로그인, 마이페이지            | 참여내역, 당첨내역, 배송정보 관리             |
| **배송 관리**         | 운영자용 대시보드                      | 당첨자 배송지 확인 및 배송완료 처리           |

| 페이지                      | 주요 요소                               |
| --------------------------- | --------------------------------------- |
| **홈(Home)**                | 진행 중/마감된 추첨 리스트, 공정성 소개 |
| **추첨 등록 페이지(Admin)** | 상품 등록 폼, 시드 소스 선택            |
| **추첨 상세 페이지**        | 참여 버튼, 참여 조건, 시드 정보         |
| **결과 확인 페이지**        | 당첨자 공개, 검증 코드 보기             |
| **회원 페이지(My)**         | 참여 내역, 배송 정보 입력               |
| **관리자 대시보드**         | 상품/회원/배송 관리 UI                  |

- **기능:**
  - **반응형 디자인:** 모바일, 태블릿, 데스크톱 등 다양한 기기에서 잘 보이도록.
  - **디자인**: Clean & Modern Minimalism과 Bold & Expressive Typography를 적절히 사용할것.
  - **다크 모드**: 다크모드, 라이트모드, 시스템 모드를 선택할 수 있도록.
  - **간단한 애니메이션/인터랙션:** 사용자 경험을 향상시킬 수 있는 요소 (스크롤 애니메이션, 호버 효과 등).
  - **SEO 친화적:** 검색 엔진에 노출이 잘 되도록 기본적인 메타태그 설정 등.

### 3. **콘텐츠 요구사항**

#### 📄 텍스트 콘텐츠

- 브랜드 메시지: “누구나 검증 가능한 추첨”
- 소개 문구: “믿지 않아도 됩니다. 직접 확인하세요.”
- 공정성 설명 콘텐츠: 시드·해시·오픈소스 원리 소개 섹션 필요
- 개인정보 처리 안내: 회원정보, 배송정보 관련 안내문 필수
  - 시즌, 이벤트에 따라 디자인 시스템을 변경하여 사용자에게 사이트에서 다른 느낌을 제공할 수 있게한다.
    - 기본 디자인 시스템, 할로윈 디자인 시스템, 크리스마스 시스템 등.
  - 페이지를 만들 경우, 스크롤이 반드시 필요 없는 화면에서는 한 화면 내(스크롤 x)에 사용자들에게 UI가 보여질 수 있도록 한다.

#### 🖼 이미지/시각 자료

- 추첨 과정 다이어그램(시드 → 난수 → 결과 재현)
- 각 추첨 상품 이미지 및 브랜드 로고
- 결과 페이지 내 검증 UI (seed, 참여자 목록, 결과 버튼)

#### 📈 외부 데이터 연동

- 주식시장 종가 API, 기상청 데이터 등 외부 시드 소스
- 추후 오픈소스 코드 링크 (GitHub 등)

#### 기타

- 이미지 or 비디오:\*\* 저작권에 위배되지 않도록 https://pixabay.com 에서 적절한 것을 사용할 것.
- **아이콘:** Heroicons 아이콘 팩을 사용할 것.

### 4. **성능 및 접근성**

- **로딩 속도:** 빠른 페이지 로딩 속도 (Core Web Vitals 고려).
- **접근성 (Accessibility):** 시각 장애인 등 다양한 사용자를 위한 웹 접근성 고려 (ARIA 속성, 키보드 내비게이션 등)

#### ⚡ 성능 목표

| 항목              | 기준                                      |
| ----------------- | ----------------------------------------- |
| **LCP**           | 2.0초 이하 (SSR + CDN 활용)               |

| **이미지 최적화** | Next.js Image + WebP 변환                 |
| **데이터 무결성** | 시드값 검증 및 해시 일치 확인 로직        |

- 색상 대비율 4.5:1 이상 유지 (공정성 강조 UI는 대비 높게)
- 키보드 접근 가능한 버튼/폼 구조
- 스크린 리더용 ARIA Label 제공
- 모바일·PC 반응형 지원 (최소 해상도 360px)

- 프레임워크는 next.js app router 최신 버전으로 만들것.
- 프로젝트 구축 시 CLI에서 상호작용이 있으면 모두 기본 옵션 즉, CLI에서 기본으로 활성화된 옵션을 사용해서 프로젝트를 구축할 것.
  - 비대화형 플래그(eg, `web`)를 사용해서 구축해 나갈 것
- 컴포넌트 기능을 최대한 활용하며 모던한 프론트엔드 개발자들이 사용하는 설계 패턴을 사용하여 개발할 것.

### 9. **프로젝트 규칙**

- #### 기술 스택
  - Framework: Next.js 14 (App Router)
  - Language: TypeScript
  - Styling: Tailwind CSS
  - Data Fetching: React Query + ky
  - Form: React Hook Form + Zod
  - State: Jotai (필요시)
  - Formatting: ESLint + Prettier
  - Git Hooks: Husky

- #### 폴더 구조

  src/
  ├── app/ # Next.js 페이지
  ├── components/ # 컴포넌트 (common, features)
  ├── hooks/ # 커스텀 훅 (React Query 포함)
  ├── lib/ # API, 스키마, 상수, 유틸
  └── types/ # 타입 정의

- ### 코딩 컨벤션
  - #### 파일명
    - 컴포넌트: PascalCase (Button.tsx)
    - 훅: camelCase (useRaffles.ts)
    - 유틸/API: camelCase (api.ts)
    - 타입: camelCase (index.ts)

  - #### 컴포넌트 작성
    - 함수형 컴포넌트만 사용
    - export function 사용 (export default 지양)
    - Props는 interface로 정의
    - 'use client' 필요한 경우에만 명시
      ```
      예시:
      interface ButtonProps {
        children: React.ReactNode;
        onClick?: () => void;
      }
      export function Button({ children, onClick }: ButtonProps) {
      return <button onClick={onClick}>{children}</button>;
      }
      ```

- #### API 작성
  - lib/api.ts에 모든 API 함수 작성
  - ky 사용
  - 타입 명시
    ```
    예시:
    export const raffleApi = {
      getList: () => api.get('raffles').json<RaffleList>(),
      getById: (id: string) => api.get(`raffles/${id}`).json<Raffle>(),
    };
    ```

- #### 커스텀 훅
  - use로 시작
  - React Query 사용시 queryKey 명확하게
  - 타입 제네릭 활용
    ```
    예시:
    export function useRaffles() {
      return useQuery<RaffleList>({
        queryKey: ['raffles'],
        queryFn: () => raffleApi.getList(),
      })
    ```

- #### 스타일링
  - Tailwind CSS 유틸리티 클래스만 사용
  - cn() 함수로 조건부 클래스 처리
  - 인라인 스타일 지양

- #### 타입 정의
  - interface 우선 사용
  - API 응답 타입은 types/index.ts에 정의
  - Zod 스키마는 lib/schemas.ts에 정의

- #### 금지 사항
  - localStorage, sessionStorage 사용 금지 (React state 사용)
  - any 타입 사용 최소화
  - console.log는 개발 중에만, 커밋 전 제거
  - 하드코딩된 문자열/숫자는 constants.ts로 분리

- #### Git 커밋 메시지
  - feat: 새 기능
  - fix: 버그 수정
  - refactor: 리팩토링
  - style: 스타일 변경
  - docs: 문서 수정
  - chore: 기타 변경
    `예시: feat: 추첨 목록 조회 기능 추가`
